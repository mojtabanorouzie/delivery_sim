# ADR-002: Couriers as time-parameterised trajectories

**Status:** Accepted — implemented in step 3b  
**Layer:** Domain Entities

---

## Context

The original `BikeCourier` held mutable `_x`, `_y` position state and advanced it
each tick via `step(dt, sim_time)`.  The Simulator detected arrival by polling
`has_target` before and after every `step()` call.

This design has several problems:

1. **Tick-rate coupling.** Movement accuracy depended on `dt`; large ticks caused
   spatial quantisation errors.  Clamping logic in `step()` patched this but added
   complexity.
2. **Arrival polling is fragile.** The `has_target` before/after pattern requires the
   Simulator to call `step()` exactly once per tick and then inspect state changes —
   a hidden protocol that breaks if anything calls `step()` out of order.
3. **No analytic position.** There was no way to ask "where will this courier be at
   `t = 47.3 s`?" without simulating every tick up to that point.

## Decision

Couriers are **time-parameterised trajectories**.  After `assign()` is called, the
entire trajectory is fully determined; no further external mutation is required.

- `assign(order_id, store_id, sim_time, target_x, target_y)` establishes the
  trajectory: it records the origin (current `_origin_x/_origin_y`), the assignment
  time, the target, and computes `_arrival_time_val` via
  `routing.travel_time(origin, target, speed)`.
- `position_at(t: float) -> tuple[float, float]` returns position at any absolute
  simulation time `t`:
  - `t <= assign_time` → origin
  - `t >= arrival_time` → target (clamped, no overshoot)
  - otherwise → linear interpolation at constant speed
- `arrival_time() -> float | None` exposes the pre-computed arrival timestamp.
  The Simulator schedules a `COURIER_ARRIVED` event at this time; it does **not**
  poll `has_target` to detect arrival.
- `has_target` is demoted to a **status predicate** (busy vs. idle) for snapshots
  and rendering.  It is True from `assign()` until `reset()`, regardless of whether
  `arrival_time` has passed.
- `step(dt, sim_time)` is **removed** with no compatibility shim.
- `x` and `y` as public Courier ABC properties are **removed**.  The canonical
  position query is `position_at(t)`.  `CourierSnapshot.x/y` in
  `render/protocol.py` is a separate DTO populated by the Simulator calling
  `position_at(clock.elapsed)`.

### Multi-leg journeys and explicit leg-start (amendment — step 3b)

The original decision assumed `_origin_x/_origin_y` would double as the trajectory
start.  This was rejected because the two-leg journey (origin → store → customer)
requires a second `assign()` call whose start point is the store location, not the
spawn origin.

**Amended contract:**

- `assign()` takes two additional parameters: `from_x: float, from_y: float`.  The
  Simulator supplies both endpoints of every leg explicitly.
- `_start_x/_start_y` (new internal fields) record the leg start from `from_x/from_y`
  at `assign()` time.  `position_at()` interpolates between `(_start_x, _start_y)`
  and `(_target_x, _target_y)`.
- `_origin_x/_origin_y` is now **strictly the spawn/idle position** set by the
  constructor and `reset()` (episode boundary).  It is never read during trajectory
  math.  The two fields are distinct concepts; do not conflate them.
- `reset(origin_x, origin_y)` is the **episode-boundary operation only**.  It is
  **not called between legs** mid-episode.  The Simulator passes the correct
  `from_x/from_y` to each `assign()` call; `reset()` is never a leg-transition tool.

**SETTLED precondition (amendment — step 3b):**

`assign()` may only be called when the courier is *settled*:
- `arrival_time()` is `None` (no prior leg), or
- `sim_time >= arrival_time()` (prior leg complete).

Calling `assign()` while `sim_time < arrival_time()` raises `ValueError`.  This
prevents silent teleportation of a moving courier.  The two-leg journey is safe
because leg 2 is only dispatched from the `ORDER_READY` handler, which fires after
the `COURIER_ARRIVED_STORE` event — guaranteeing `sim_time >= arrival_time()`.

**Trajectory determinism (amended wording):**

Trajectory is deterministic: same `(from_x, from_y, target_x, target_y, assign_time,
speed)` → identical `position_at` samples.

### Routing model

`assign()` uses `self._routing.travel_time(...)` exclusively — never inline
`distance / speed`.  This keeps the travel-time calculation behind the
`RoutingModel` ABC so that a road-graph model (step 8) can return a different
travel time without changing `BikeCourier`.

## Consequences

**Positive:**
- Courier arrival time is computed once at dispatch and is exact — no tick-rate
  quantisation.
- The Simulator can schedule `COURIER_ARRIVED` events without any per-tick polling.
- `position_at(t)` enables analytic snapshot production at any time.
- Trajectory is deterministic: same `(origin, target, assign_time, speed)` →
  identical `position_at` samples.

**Negative / watch-outs:**
- The straight-line interpolation in `position_at` is valid only for Euclidean
  routing.  When a road-graph routing model is introduced (step 8), `position_at`
  may need to walk waypoints rather than using a single linear fraction.  The
  `RoutingModel.route()` method already exposes waypoints for this purpose.
- `WorldState.snapshot()` (step 3c) must call `position_at(clock.elapsed)` to
  populate `CourierSnapshot.x/y`; it must not read any property named `x` or `y`
  directly from the courier.

## Shipped contract (`entities/courier.py`)

```python
class Courier(ABC):
    # Removed: x/y properties, step(dt, sim_time)
    def position_at(self, t: float) -> tuple[float, float]: ...
    def arrival_time(self) -> float | None: ...
    def assign(self, order_id, store_id, sim_time, target_x, target_y) -> None: ...
    def reset(self, origin_x: float, origin_y: float) -> None: ...
    def has_target(self) -> bool: ...  # status predicate, NOT arrival path

class BikeCourier(Courier):
    # Internal: _origin_x/_origin_y (spawn + trajectory start), _assign_time,
    # _target_x/_target_y, _arrival_time_val.  All set by assign(); cleared by reset().
```
