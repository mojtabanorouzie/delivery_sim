# ADR-001: Event-driven clock as source of truth

**Status:** Accepted — implemented in step 3b  
**Layer:** Simulation Engine

---

## Context

The original `SimClock` advanced time by fixed increments (`advance()` → `_tick += 1`,
`elapsed = _tick * dt`).  The Simulator loop called `advance()` once per iteration,
making `dt` the resolution of all temporal reasoning.  This creates two problems:

1. **Precision loss.** Any event that does not fall on a `dt` boundary (e.g. a courier
   arriving at `t = 7.3 s` when `dt = 1.0`) is either rounded or requires sub-tick
   interpolation scattered across the engine.
2. **Wasted iterations.** When nothing happens between `t = 5` and `t = 100`, the loop
   still executes 95 ticks, calling all entities each time with nothing to do.

## Decision

Replace fixed-increment advance with **event-driven advance**:

- `SimClock.advance_to(t: float)` jumps the clock to absolute simulation time `t`,
  where `t` is the time of the next event popped from the `EventQueue`.
- `SimClock.elapsed` returns the current time directly (`_time`); it is not computed
  from a tick counter.
- **Ticks are observers, never the source of truth.** `SimClock.tick` counts
  `advance_to` calls and is available for renderers and metrics collectors that want
  a step counter; it does not govern advance size.
- `dt` is retained on `SimClock` as an **optional observer / sampling cadence**
  (e.g. for fixed-rate renderers).  It is stored on the clock for convenience but
  plays no role in `advance_to` and is not used to compute step sizes.  It remains
  in `ScenarioConfig` / YAML as a documented sampling parameter.

## Consequences

**Positive:**
- Courier arrival, order creation, and any future event can be scheduled and
  processed at exact floating-point times with no rounding.
- Episodes with sparse events are fast: the loop only runs when something happens.
- The Simulator loop becomes `while events: clock.advance_to(queue.pop().time); process(event)`.

**Negative / watch-outs:**
- Renderers that expect fixed-cadence snapshots must either schedule their own
  SNAPSHOT events at `dt` intervals or accept variable-interval snapshots.
- `dt` in the YAML may become vestigial if no renderer or reward function uses it;
  review at step 6.

## Shipped contract (`engine/clock.py`)

```python
class SimClock:
    def __init__(self, dt: float = 1.0) -> None: ...  # dt = observer cadence
    @property
    def elapsed(self) -> float: ...      # current sim time
    @property
    def tick(self) -> int: ...           # advance_to call count (observer)
    def advance_to(self, t: float) -> None: ...  # jump to t; raises if t < elapsed
    def reset(self) -> None: ...         # elapsed=0, tick=0
```

`advance()` (fixed-increment) is removed with no compatibility shim.
